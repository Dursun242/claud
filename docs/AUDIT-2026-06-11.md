## 🔴 Critique

### 1. `proces_verbaux_reception` n'a pas de RLS

`db-migrations/001_procès_verbaux_reception.sql` crée la table **sans** `ENABLE ROW LEVEL SECURITY` ni aucune policy (aucun fichier SQL du repo n'en ajoute). Sur Supabase, une table publique sans RLS est lisible/modifiable par **tout utilisateur authentifié** via PostgREST avec la clé anon — et `/api/pv-reception/list/route.js:46` utilise `userClientFromToken()` en comptant justement sur le RLS pour filtrer.

**Conséquence** : n'importe quel client MOA peut lire (et modifier) les PV de réception de **tous** les chantiers, y compris emails des signataires, statuts, motifs de refus — directement via l'API Supabase, sans même passer par les routes `/api/pv-reception/*`.

**Correction** (nouvelle migration `db-migrations/003`) :

```sql
ALTER TABLE proces_verbaux_reception ENABLE ROW LEVEL SECURITY;

CREATE POLICY "pv_select" ON proces_verbaux_reception FOR SELECT TO authenticated
  USING (public.is_staff() OR public.client_has_chantier(chantier_id));

CREATE POLICY "pv_write_staff" ON proces_verbaux_reception
  FOR ALL TO authenticated
  USING (public.is_staff()) WITH CHECK (public.is_staff());
```

À appliquer en priorité absolue. Vérifier dans le dashboard Supabase que la table est bien actuellement sans RLS (`select relrowsecurity from pg_class where relname = 'proces_verbaux_reception';`).

## 🟠 Élevé

### 2. IDOR sur `/api/upload` — pas de vérification d'ownership

`src/app/api/upload/route.js:31-70` : après `verifyAuth()`, la route insère dans `attachments` via le **service role** (bypass RLS) avec `type` et `itemId` fournis par le client, sans vérifier que l'utilisateur a accès à ce chantier/OS/CR/tâche. Un client MOA peut uploader des fichiers sur n'importe quel chantier. De plus, `type` n'est pas validé avant la construction du chemin Storage (`${type}/${itemId}/...` ligne 55) : un `type` arbitraire crée des fichiers orphelins à des chemins non prévus (l'insert DB échoue ensuite, mais le fichier Storage reste).

**Correction** : valider `type` contre `['chantier','os','cr','task']` en début de route, puis vérifier l'accès avec un client porteur du token utilisateur (`userClientFromToken`) avant l'upload :

```js
const VALID_TYPES = { chantier: 'chantiers', os: 'ordres_service', cr: 'compte_rendus', task: 'taches' }
if (!VALID_TYPES[type]) return Response.json({ error: 'Type invalide' }, { status: 400 })

const userSupa = userClientFromToken(extractBearerToken(request))
const { data: parent } = await userSupa.from(VALID_TYPES[type]).select('id').eq('id', itemId).maybeSingle()
if (!parent) return Response.json({ error: 'Ressource non accessible' }, { status: 403 })
```

### 3. Next.js 15.5.15 vulnérable (npm audit : high)

Série d'avis sur la plage installée : bypass middleware/proxy (GHSA-26hh-7cqf-hhc6, GHSA-492v-c6pp-mqqv…), cache poisoning RSC, XSS via nonces CSP, DoS image optimization, SSRF WebSocket. Corrigé en **15.5.19**.

**Correction** : `npm install next@15.5.19` puis `npm test && npm run build`. Traiter aussi le reste : `npm audit fix` (ws, brace-expansion, @tootallnate/once).

### 4. Pas de rate limit sur `/api/ai-qonto`

`src/app/api/ai-qonto/route.js` appelle Claude (coûteux) après fetch Qonto, avec `verifyAuth()` mais **sans** rate limiting, contrairement à `/api/claude` (20/min/IP) et `/api/extract-*` (10/min/IP). Un utilisateur authentifié peut drainer le quota Anthropic.

**Correction** : répliquer le pattern de rate limit existant (5–10 req/min/IP). À terme, factoriser ce pattern dupliqué dans `lib/rateLimit.js` (voir point 10).

## 🟡 Moyen

### 5. Policies « Allow all » résiduelles (migration 002)

`migrations/002_v3_validation_os_cr_interactif.sql:113-115` crée `os_validations`, `cr_commentaires`, `chantier_photos` avec `USING (true) WITH CHECK (true)`. Les migrations 004/005 ne les corrigent **pas**. Ces tables ne sont référencées **nulle part** dans `src/` (code mort), donc probablement vides — mais elles restent ouvertes à tout utilisateur authentifié.

**Correction** : si elles sont vides et inutilisées → migration qui les `DROP`. Sinon → policies par chantier (via jointure `ordres_service`/`compte_rendus`).

Même remarque pour les tables de `migrations/001_refonte_complete.sql` (`user_roles`, `plans`, `photo_reports`, `photos`) : système de rôles `user_roles` parallèle à `authorized_users`, jamais utilisé dans `src/`. À déprécier/dropper pour éliminer la confusion.

### 6. Fallback RLS par prénom toujours actif (migration 019)

`client_has_chantier()` matche par `client_user_id` (UUID, robuste) avec fallback `client ILIKE '%prénom%'` quand `client_user_id IS NULL`. Deux clients homonymes peuvent encore se voir mutuellement les chantiers non backfillés.

**Vérification en prod** : `SELECT COUNT(*) FROM chantiers WHERE client_user_id IS NULL;` — si 0, créer la migration qui supprime le fallback et durcit la fonction.

### 7. Erreurs Supabase avalées dans `shared.js`

`deleteCR()`/`deleteOS()` (`src/app/dashboards/shared.js` ~342, ~392) logguent l'erreur en console et retournent sans signal : si le delete échoue (RLS, réseau), l'UI retire l'élément alors qu'il est toujours en base. Retourner `{ ok, error }` (ou `throw`) et afficher un toast côté appelant.

### 8. SDK Anthropic obsolète

`@anthropic-ai/sdk ^0.20.0` est très ancien. Mettre à jour vers la dernière version stable (vérifier les breaking changes sur les appels dans `/api/claude`, `/api/extract-*`, `/api/ai-qonto`). Faire un `npm outdated` général au passage.

### 9. Validation d'input à renforcer sur quelques routes

- `/api/pappers` : valider `siret` (`/^\d{14}$/`) et borner la longueur de `q` avant de proxyfier vers Pappers.
- `/api/pv-reception/create` : valider format/longueur des emails signataires avant de les envoyer à Odoo Sign.
- `/api/pv-reception/decision` : vérifier que l'appelant est l'un des signataires du PV avant d'enregistrer la décision.
- `/api/odoo/sync-signatures` : valider le statut retourné par Odoo contre une liste blanche avant écriture en base.

### 10. Pattern rate-limit dupliqué et fragile

La `Map` + `setInterval` module-level est copiée dans `/api/claude` et `/api/extract-*` : timers multiples (un par worker), pas de borne de taille de la Map. Factoriser dans `lib/rateLimit.js` avec garde singleton et `maxEntries` ; l'utiliser aussi pour `/api/ai-qonto` (point 4).
